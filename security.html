How we think about security

Small surface area. Clear evidence. Controlled side effects.

Primary sources

Source systems remain primary

Slack, Microsoft Teams, Jira, GitHub, and uploaded documents stay the source of truth. Verachi links back to those systems instead of pretending to replace them.

Evidence first

Citations are part of the product

AI summaries and chat answers must point back to concrete sources. If Verachi cannot cite a claim, it should say so plainly.

Least privilege

Write-backs are conservative

External write-backs are optional, controlled, and audit-backed. Verachi reads more than it writes, and sensitive actions are designed to be reviewable.

Data handling

What Verachi stores.

The default integration model is pointer-only: Verachi stores its own decision record and source pointers, not raw third-party archives.

Stored

Durable Verachi records

  • Decision titles, summaries, rationale, status, owners, and project links
  • Risk findings, guideline evaluations, and user-visible audit events
  • Source links, provider IDs, routing IDs, counts, labels, status, and detection metadata
  • Uploaded documents and chat messages that users explicitly put into Verachi
Not durable by default

Raw external content

  • Slack or Teams thread bodies and replies
  • Jira issue descriptions, comments, and linked issue bodies
  • GitHub PR bodies, issue bodies, review comments, repository contents, and patches
  • External artifact snapshots for Slack, Jira, and GitHub store source pointers, not content snapshots
Controls

Protection layers in the current platform.

Tenant isolation

Database boundaries

Tenant-scoped tables use Postgres Row-Level Security, and application code sets tenant context per request. API routes enforce authenticated access before returning customer data.

Encryption

At rest and in transit

PostgreSQL storage and object storage use infrastructure-level encryption at rest. Web, S3 SDK, and database traffic use encrypted transport. Integration secrets are encrypted before storage.

Auditability

Inspectable actions

Syncs, sensitive actions, AI-generated summaries, and external write-backs are designed to leave audit evidence so workspace members can inspect what happened and why.

AI grounding

Context, not general guessing

Ask Verachi operates over workspace, project, decision, document, and source context. Factual claims should include citations to Slack, Jira, GitHub, uploaded documents, or internal records.

Access keys

Scoped interfaces

API keys are hashed and scoped. MCP is supported through a stateless HTTP endpoint for selected project, decision, document, guideline, and risk workflows.

No surveillance

Decision memory, not productivity tracking

Verachi is built around decisions, risks, guidelines, and evidence. It does not create individual productivity scores, keystroke monitoring, or manager dashboards comparing people.

Integrations

How connected tools are used.

When needed, integrations read source content temporarily for routing, analysis, and risk evaluation. Durable storage follows the pointer-only model unless a user explicitly uploads or enters content into Verachi.

Compliance status

Clear about today's controls and the next steps.

Implemented controls

SOC 2 readiness foundations

Encryption at rest, encryption in transit, logical tenant isolation, authenticated access checks, and audit logging are documented as current controls. Verachi itself is not presented as SOC 2 certified.

In progress

GDPR and data lifecycle

Data minimization and confidentiality controls are strong, including pointer-only integration storage. We are expanding lifecycle documentation and customer-facing privacy workflows for retention, deletion, data-subject requests, and processor records.

Roadmap

Operational hardening

We are continuing to mature the security program with clearer lifecycle controls, documented key rotation, managed secrets, storage access evidence, vendor records, and independent review ahead of a future Type II audit window.

Special requirements

Confirm regulated use cases

For HIPAA, strict data residency, BAA, or certification-bound workloads, we review requirements with the customer before production use.

Infrastructure

Built on providers with mature security programs.

FAQ

Common security questions.

Does Verachi store Slack, Teams, Jira, or GitHub message bodies?

Not as durable storage by default. External content can be fetched temporarily for analysis, routing, and risk evaluation, but durable records store Verachi-authored conclusions, links, IDs, counts, labels, status, and detection metadata.

What context is sent to AI?

No. AI operates on the context needed for a requested summary, answer, decision capture, or risk evaluation. Claims should be grounded with citations, and raw integration content is not kept as a permanent archive by default.

Can Verachi write back to my tools?

Only through explicit user action or configured workspace rules. External write-backs are conservative, non-destructive where possible, and audit-backed.

Is Verachi SOC 2 certified?

No. Verachi has documented SOC 2 readiness controls and a hardening roadmap, but it should not be represented as SOC 2 certified.
